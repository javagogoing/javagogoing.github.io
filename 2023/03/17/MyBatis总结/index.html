MyBatis总结

发表于2023-03-17|更新于2023-03-17
|阅读量:

MyBatis是什么

MyBatis是一款优秀的持久层框架，一个半自动ORM（对象关系映射）框架，它支持定制化SQL、存储过程以及高级映射。MyBatis几乎避免了所以的JDBC代码和手动设置参数以及获取结果集。MyBatis可以使用简单的XML或注解来配置和映射原生类型、接口和java的POJO（Pliain Old Java Objects，普通老式 java 对象）为数据库中的记录

什么是“持久化”

持久（Persistence），即把数据（如内存中的对象）保存到可以永久保存的存储设备中（如磁盘）。持久化的主要应用是将内存中的数据存储在关系型的数据库中，当然也可以存储在磁盘文件中、XML数据文件中等等。

什么是“持久层”

持久层（Persistence Layer）即专注于实现数据持久化应用领域的某个特定系统的一个逻辑层面，将数据使用者和数据实体相关联。

什么是ORM

对象关系映射（英语：Object Relational Mapping，简称ORM，或O/RM，或O/R mapping），是一种程序设计技术，用于实现面向对象编程语言里不同类型系统的数据之间的转换。从效果上说，它其实就是创建了一个可在编程语言里使用的“虚拟对象数据库”。

ORM是面向对象程序设计语言和关系型数据库发展不同步时的解决方案，采用ORM框架后，应用程序不再直接访问底层数据库，而是以面向对象的方式来操作持久对象，而ORM框架则将这些面向对象的转换成底层的SQL操作。

为什么要做持久化和ORM设计

在目前的企业应用系统设计中，MVC，即Model（模型） -View（视图） -Controller（控制）为主要的系统架构模式。MVC中的Model包含了复杂的业务逻辑和数据逻辑，以及数据库存取机制（如JDBC的连接、SQL生成和Statement创建、还要ResultSet结果集的读取等）等。这些复杂的业务逻辑和数据逻辑分离，以将系统的紧耦合关系转化为送耦合关系（即解耦合），是降低系统耦合度迫切要做的，也是持久化要做的工作。MVC模式实现了架构上的表现层（即View）和数据处理层（即Model）分离的解耦合，而持久化的设计则实现了数据处理层内部的业务逻辑和数据逻辑分离的解耦合。而ORM作为持久化设计中的最重要也是最复杂的技术，也是目前业界热点技术。

简单来说，按通常的系统设计，使用JDBC操作数据库，业务处理逻辑和数据处理逻辑是混杂在一起的

  • 一般来说都是如下几个操作：
    1. 建立数据库连接，获得Connection对象。
    2. 根据用户的输入组装查询SQL语句。
    3. 根据SQL语句建立Statement对象或者PreparedStatement对象。
    4. 用Connection对象执行SQL语句，获得结果集ResultSet对象。
    5. 然后一条一条读取结果集ResultSet对象中的数据。
    6. 根据读取到的数据，按特定的业务逻辑进行计算。
    7. 根据计算得到的结果再组装更新SQL语句
    8. 在使用Connection对象执行更新SQL语句，以更新数据库中的数据。
    9. 最后依次关闭各个Statement 对象和 Connection对象。

MyBatis的实现机制

  1. 读取MyBatis的全局配置文件MyBatis-config.xml
    MyBatis-config.xml文件包括一系列配置信息，其中包括标签，此标签配置了映射节点，映射节点内部定义了SQL语句
  2. 创建SqlSessionFactory会话工厂
  3. 创建SqlSession会话
  4. 执行查询操作

什么是一对一映射关系？

从数据库的角度出发就是在任意一个表中引入另外一个表的主键作为外键。在本类中定义另外一个类的对象。
在mybatis中，通过resultMap元素的子元素association来处理

  • association中的配置属性：

    • property：指定映射到的实体类对象属性，与字段一一对应

    • select：指定引入嵌套查询的子SQL语句，该属性用于关联映射中的嵌套查询。

    • javaType：指定映射到实体对象属性的类型

什么是一对多关系？

一对多就是表A中一条数据对应表B中多条数据，比如：用户和订单之间的关系，一个用户可以有多个订单信息。
在mybatis中通过resultMap元素的子元素collection来进行处理。

  • collection中的配置元素：

    • property：指定映射到实体类对象属性，与表字段一一对应

    • column：指定表中对应的字段

    • select：指定引入嵌套查询的子SQL语句，该属性用于关联映射中的嵌套查询。

    • javaType：指定映射到实体对象属性的类型

    • ofType：ofType属性与javaType属性对应，它用于指定实体对象中集合类属性所包含的元素类型。

什么是多对多关系？

多对多关系就是表A中一条数据对应表B中多条数据，表B中一条数据也可以对应表A中多条数据，比如，用户可以对应多个角色，角色也可以对应多个用户

MyBatis中的缓存

  1. MyBatis的一级缓存（默认开启）

在同一个Session中执行相同的SQL语句，就会从缓存中直接获取，不会从数据库重新查询

  • 使一级缓存失效的四种情况：

    • 不同的SqlSession对应不同的一级缓存

    • 同一个SqlSession但是查询条件不同

    • 同一个SqlSession两次查询期间执行了任何一次增删改操作

    • 同一个SqlSession两次查询期间手动清空了缓存

  1. MyBatis的二级缓存

    • 二级缓存是SqlSessionFactory级别，通过同一个SqlSessionFactory创建的SqlSession查询的结果会被缓存；此后若再次执行相同的查询语句，结果就会从缓存中获取

    • SqlSessionFactory有多个方法创建SqlSession实例。
      常用的两个方法：

      • openSession()：会默认开启一个事务，但事务不会自动提交，也就意味着需要手动提交该事务，更新操作数据才会持久化到数据库中。

      • openSession(boolean autoCommit)：参数是否自动提交，如果设置为true，那么就不要手动提交事务。

    • 二级缓存开启条件：

      • 在核心配置文件中，中设置全局配置属性cacheEnabled=”true”，默认为true，不需要设置
      • 在映射文件中设置标签
      • 二级缓存必须在SqlSession关闭或提交之后有效
      • 查询的数据所转换的实体类类型必须实现序列化的接口

    • 使二级缓存失效的情况：

      两次查询之间执行了任意的增删改，会使一二级缓存同时失效

分页查询插件

MyBatis可以使用第三方的插件来对功能就行扩展，分页助手PageHelper是将分页的复杂操作进行封装，使用简单的方式即可获得分页的相关数据

  1. 开发步骤：

    • 导入通用PageHelper的坐标
    • 在mybatis核心配置文件中配置PageHelper插件
    • 测试分页数据获取

  2. PageHelper分页原理：

    PageHelper分页的实现原理就是通过拦截器拦截执行的sql，然后再重写sql，添加物理分页语句和物理分页参数

  3. MyBatis中的执行器：

    • SimpleExecutor：简单执行器，每次执行一个update或select，就会开启一个statment对象，用完就关闭

    • ReuseExecutor：重用执行器，执行update或select，以sql作为key查询statment对象，存在就用，不存在就创建，用完后不会关闭

    • BatchExecutor：批处理执行器，执行update，将所有的sql都添加到批处理中，等待统一执行

    MyBatis中默认使用的是SimpleExecutor执行器，如果需要更改，可选择局部更改或全局更改

延迟加载

  1. 延迟加载原理
    延迟加载是开发过程中灵活获取对象的一种求值策略，该策略在定义目标对象并并不会立即计算实际对象值，而在该对象被后续实际调用时才会求值。在计算机科学中，延迟加载对应一个专业的术语：惰性加载
    简单来说就是：查询一个对象时，它相关联的对象的查询尽量往后延迟时查询，延时到我们在程序需要访问相关的对象时，再查询相关联对象。

  2. 局部延迟加载

    在association、collection标签有一个fetchType属性，该属性的默认值为eager（立即加载）fetchType=”eagr”，如果想用延迟加载设置fetchType=”Lazy”

  3. 全局延时加载

    在核心配置文件中settings标签中的setting标签进行设置，所有的分表都使用延迟加载，如果不想使用延迟加载，就单独加fetchType=”eager”

  4. 为什么要使用延迟加载

    其实是为了提高数据库的访问效率，因为往往设计到多表查询的时候，这样影响查询效率，所以引入了延迟加载，提高执行效率，来实现优化性能的目的，因为查询的表越少，效率越高
